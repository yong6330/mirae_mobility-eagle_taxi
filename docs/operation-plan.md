# Mirae Mobility 프로젝트 운영 계획(안) v0.4

## 1. 문서 목적

본 문서는 Mirae Mobility 팀의 컴퓨팅사고 팀 프로젝트 「독수리 택시」 수행을 위한 운영 기준을 정리한 문서이다.

본 운영 계획안은 팀원들이 프로젝트 수행 과정에서 다시 “누가 무엇을 해야 하는지”, “어떤 문서를 어느 수준으로 작성해야 하는지”, “산출물을 어떤 기준으로 점검해야 하는지”를 재논의하느라 시간을 낭비하지 않도록, 팀 구성, 역할 분담, 주차별 업무, 산출물 관리, 기준 문서 작성 방식, 회의 운영, AI 활용 원칙을 명확히 정리하는 것을 목적으로 한다.

실제 코드 작성 순서, 폴더 구조, 데이터베이스 테이블, API 상세 구현, 프론트엔드·백엔드 개발 순서, 실행 방법, 배포 방식은 별도의 「Mirae Mobility 프로젝트 개발 계획(안)」에서 다룬다.

## 3. 프로젝트 정의

독수리 택시는 연세대학교 미래캠퍼스 학생들이 출발지, 도착지, 희망 출발 시간을 등록하고 유사한 이동 조건을 가진 학생들과 택시 파티를 구성할 수 있도록 돕는 캠퍼스 특화 택시 합승 중개 플랫폼이다.

본 프로젝트는 장기적으로 택시 호출 서비스까지 확장할 수 있는 방향성을 가지지만, 프로젝트 MVP에서는 실제 택시 호출·결제·배차 기능은 구현하지 않는다. 대신 회원가입, 로그인, 파티 생성, 파티 조회, 파티 참여, 예상 요금 분할, 파티 상태 관리, 지도 API 기반 위치 표시, 파티별 채팅 기능을 중심으로 구현한다.

| 신고 버튼 | 신고 저장 API는 구현하지 않되, 신고하기 버튼을 통해 안전 안내 또는 추후 기능 안내 모달을 표시 |

## 14. 기능명세서 작성 기준 및 예시

### 14-1. 기능명세서 작성 목적

기능명세서는 “무엇을 만들 것인지”를 기능 단위로 정의하는 문서이다.

단순히 기능명을 나열하는 것이 아니라, 기능 ID, 화면 ID, 카테고리, 기능 깊이, 중요도, 권한, 입력값, 출력값, 처리 조건, 예외상황, 연결 API까지 정리해야 한다.

이 문서를 기준으로 TL은 백엔드 기능을 구현하고, QA&Front는 화면과 테스트 항목을 구성한다.

### 14-2. 기능명세서 작성 양식 및 예시

위 하이퍼링크 클릭 시 양식 및 예시 파일로 이동합니다. 기능명세서 시트 참고바랍니다.

### 14-3. 기능명세서 작성 시 주의사항

· 기능은 반드시 Category → Depth1 → Depth2 → Depth3 구조로 분류한다.
 · 기능 ID는 F-AUTH, F-PARTY, F-MAP, F-CHAT, F-ADMIN처럼 영역별로 구분한다.
 · 중요도는 상 / 중 / 하로 구분한다.
 · 이번 프로젝트에서 실제 결제, 실제 택시 호출, 소셜 로그인은 기능명세서에 포함하지 않는다.
 · 모든 핵심 데이터는 FastAPI와 SQLite를 통해 처리한다.
 · localStorage는 로그인 토큰 등 최소 용도 외에는 사용하지 않는다.
 · 기능명세서는 “개발 가능한 기능 기준”으로 작성해야 하며, 아이디어 수준의 희망 기능을 섞지 않는다.

### 15-4. API 명세서 작성 시 주의사항

· React가 직접 SQLite를 다루지 않는다.
 · 모든 데이터 생성·조회·수정은 FastAPI API를 통해 처리한다.
 · 인증이 필요한 API는 Authorization Header의 Bearer Token을 기준으로 작성한다.
 · API Endpoint는 개발 중 임의로 변경하지 않는다.
 · 변경이 필요하면 PM과 TL이 확인 후 API 명세서에 먼저 반영한다.
 · 1주차 API 명세서는 초안이고, 4주차 통합 후 최종본으로 업데이트한다.

### 16-4. 화면 구성표 작성 시 주의사항

· 화면 구성표는 디자인 문서가 아니라 개발 기준 문서이다.
 · 색상, 폰트, 장식보다 입력 요소, 버튼, 표시 데이터, 연결 API가 중요하다.
 · Figma 고품질 디자인은 만들지 않는다.
 · Google Docs 표로 작성해도 충분하다.
 · 화면에 필요한 API가 없으면 PM의 API 명세서 v0.4에 보완 요청한다.

## 17. 화면 흐름도 작성 기준 및 예시

### 17-5. 화면 흐름표 작성 시 주의사항

· 화면 흐름표는 화면의 배치가 아니라 “사용자 이동 순서”를 작성한다.
 · 예외 흐름도 반드시 작성한다.
 · 예: 로그인하지 않은 사용자가 파티 생성 클릭, 최대 인원 초과 참여, 이미 참여한 파티 재참여 등.
 · 테스트 기획서는 화면 흐름표를 기준으로 작성한다.

## 18. 알고리즘 설계 문서 작성 기준 및 예시

### 18-1. 알고리즘 설계 문서 작성 목적

알고리즘 설계 문서는 독수리 택시가 어떤 기준으로 파티를 검색·추천하고, 요금을 나누며, 상태를 변경하는지 설명하는 문서이다.

OM은 알고리즘을 문서로 정리하고, TL은 이를 바탕으로 백엔드 로직을 구현한다.

### 18-2. 알고리즘 설계 문서 기본 양식

#### 의사코드

파티 참여 요청이 들어오면

party_id에 해당하는 파티가 존재하는지 확인한다.

없으면 “존재하지 않는 파티” 오류 반환

파티 상태가 모집 중인지 확인한다.

매칭 완료 또는 취소 상태이면 참여 불가 반환

현재 사용자가 이미 해당 파티에 참여했는지 확인한다.

이미 참여했다면 “이미 참여한 파티” 오류 반환

현재 참여 인원을 계산한다.

현재 참여 인원이 최대 인원보다 작은지 확인한다.

같거나 크면 “최대 인원 초과” 오류 반환

party_members 테이블에 사용자 참여 정보를 저장한다.

참여 후 현재 인원을 다시 계산한다.

예상 요금을 현재 인원으로 나누어 1인 예상 요금을 계산한다.

현재 인원이 최대 인원과 같다면 파티 상태를 “매칭 완료”로 변경한다.

갱신된 파티 정보, 현재 인원, 1인 예상 요금, 상태를 반환한다.

| 출발 시간이 현재 이후 | 필수 조건 |
| 성별 매칭 조건 충족 | 필수 조건 |

점수가 높은 순서대로 추천 파티를 표시한다.
 시간 점수는 중복 가산하지 않고 가장 높은 조건 하나만 적용한다.
 조건에 맞는 파티가 없으면 자동 생성하지 않고, 사용자가 새 파티를 생성할 수 있도록 안내한다.

### 18-6. 알고리즘 문서 작성 시 주의사항

· “AI가 알아서 추천한다”처럼 쓰지 않는다.
 · 장소, 시간, 인원, 요금 기준을 구체적으로 작성한다.
 · 지도 API를 이용한 실제 경로 계산은 이번 MVP 범위에서 제외한다.
 · 추천 알고리즘은 단순하고 설명 가능한 방식으로 작성한다.
 · 최종 보고서에서는 이 알고리즘을 컴퓨팅사고의 “알고리즘 설계” 단계와 연결한다.

## 38. 최종 운영 기준 문장

본 팀은 PM, OM, TL, QA&Front 역할을 기준으로 프로젝트를 수행한다. PM은 운영 계획(안), 개발 계획(안), 컴퓨팅사고 정리, API 명세서 v0.4, 산출물 점검, 전체 일정 관리, 개발 지원, 발표자료, 관리자 페이지, 최종 제출물 통합을 담당한다. OM은 알고리즘 문서, 설문조사, 설문 결과 종합, 결과 보고서 작성을 담당한다. TL은 기능명세서, FastAPI 백엔드, SQLite 데이터베이스, 핵심 로직, 프론트·백엔드 통합을 담당한다. QA&Front는 화면 구성표 및 흐름표 작성, React 프론트엔드 개발, 테스트 기획 및 실행을 담당한다.

프로젝트 수행 기간에는 기획을 반복 논의하기보다 1주차에 기준문서를 작성하고, 2주차부터 프론트엔드와 백엔드 개발 및 설문조사 준비를 병행한다. 3주차에는 핵심 개발 완성과 설문 결과 종합을 목표로 하며, 4주차에는 통합, 테스트, 버그 패치, 결과 보고서, 발표자료, 관리자 페이지, API 명세서 최종 정리에 집중한다.

최종적으로 독수리 택시는 React 프론트엔드, FastAPI 백엔드, SQLite 데이터베이스를 기반으로 회원가입, 로그인, 파티 생성, 파티 조회, 파티 참여, 요금 분할, 지도 표시, 파티별 채팅 기능을 갖춘 프로젝트 MVP로 구현한다
