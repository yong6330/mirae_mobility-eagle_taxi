## 1. 용어 사전 (자주 헷갈리는 단어 정리)

| 용어 | 우리 프로젝트에서의 의미 |
|---|---|
| **FastAPI** | Python으로 API 서버를 만드는 프레임워크. 우리 백엔드의 뼈대 |
| **SQLAlchemy** | 파이썬 코드로 DB 조작하게 해주는 라이브러리. SQL 직접 안 쓰고 객체로 다룸 |
| **SQLite** | DB 서버 설치 없이 파일 하나(`eagle_taxi.db`)로 동작하는 가벼운 DB |
| **ORM** | Object-Relational Mapping. DB 테이블을 파이썬 객체처럼 다루는 방식 |
| **Pydantic** | 요청·응답 데이터의 형식·타입을 자동 검증하는 라이브러리 |
| **API** | 프론트가 백엔드에게 "이거 해줘"라고 부탁하는 통로. URL+메서드로 식별 |
| **엔드포인트(endpoint)** | 하나하나의 API 주소. 예: `POST /api/auth/register` |
| **라우터(router)** | 같은 부류의 엔드포인트를 묶은 파일. 예: 인증 라우터, 파티 라우터 |
| **스키마(schema)** | 요청·응답의 형식 정의. "이메일은 문자열, 비밀번호는 8자 이상" 같은 규칙 |
| **모델(model)** | DB 테이블 1개를 표현한 파이썬 클래스. 예: `User`, `Party` |
| **JWT (제이더블유티)** | 로그인 성공 시 발급되는 암호 출입증. 헤더에 담아 보내면 누군지 식별 |
| **Bearer 토큰** | JWT를 보내는 방식. `Authorization: Bearer <토큰>` 형태 |
| **bcrypt (비크립트)** | 비밀번호를 안전하게 해시(암호화)해서 저장하는 알고리즘 |
| **해시(hash)** | 원문을 되돌릴 수 없는 암호화 결과. 비밀번호 저장에 쓰임 |
| **CHECK 제약** | DB가 직접 거부하는 값 규칙. 코드 버그가 있어도 잘못된 값 못 들어감 |
| **CORS** | 다른 주소(예: localhost:5173)에서 우리 백엔드(localhost:8000) 부르는 거 허용하는 설정 |
| **Swagger UI (`/docs`)** | API 목록을 브라우저에서 직접 호출·테스트할 수 있는 페이지 |
| **ReDoc (`/redoc`)** | API 명세를 예쁘게 읽기만 가능한 페이지. 시연용 X |
| **KST** | 한국 시간 (UTC+9). 우리 프로젝트 모든 시간 기준 |
| **ISO 8601** | 날짜 표기 표준 형식. 예: `2026-05-13T18:00:00` |
| **Kakao Mobility Directions API** | 출발/도착 좌표 → 실제 거리·요금·시간 알려주는 카카오 외부 API |
| **Kakao Maps JavaScript API** | 프론트에서 지도 표시·장소 검색에 쓰는 카카오 외부 API. 위 거랑 별개 |
| **fallback** | "원래 방법이 안 될 때 쓰는 차선책". 우리는 Kakao Key 없을 때 요금 0원 처리하는 모드 |
| **estimated_fare** | Kakao가 알려준 예상 총 택시비 (원) |
| **per_person_fare** | estimated_fare를 현재 인원으로 나눈 1인 분담금. DB에 저장 안 하고 계산해서 응답 |
| **current_members** | 파티 현재 참여 인원수. DB에 저장 안 하고 party_members 테이블 count로 계산 |
| **gender_rule** | 파티 성별 매칭 옵션. `any` 또는 `same_gender` |
| **party_gender** | same_gender 파티의 기준 성별. 생성자 성별이 들어감 |
| **status** | 파티 상태. `recruiting`(모집중) / `matched`(정원 도달) / `canceled`(취소) / `expired`(만료) |
| **lifespan** | FastAPI 서버 시작/종료 시점에 실행되는 코드. 우리는 시작 시 DB 테이블 자동 생성에 사용 |
| **uvicorn** | FastAPI 앱을 실제로 실행하는 웹 서버 |
| **venv (가상환경)** | 프로젝트 전용 파이썬 패키지 보관소. 시스템 파이썬 안 더럽힘 |
| **pytest** | 자동 테스트 실행 도구. 우리는 27건 작성 |
| **Alembic** | DB 구조 변경 이력 관리 도구. 현재 미도입 (MVP라 생략) |
| **MVP** | Minimum Viable Product. 최소 기능 갖춘 데모용 결과물 |

## 5. 기술 배경 설명 (회의 중 질문 들어왔을 때 대처용)

### 5-1. 왜 Kakao Mobility API에 의존하는가?
택시 합승 앱에서 "5,000원 예상" 같은 숫자를 띄우려면 실제 도로 거리·교통 상황 반영 시간·한국 택시 요금 체계를 알아야 함. 우리가 다 만들 수 없으니 Kakao에 좌표 보내고 결과 받아오는 방식. 명세서 F-PARTY-005에 처음부터 그렇게 설계됨. Key 없으면 → 모름 → 0원 표시 (fallback).

### 5-2. 왜 SQLite를 쓰는가?
별도 DB 서버 설치 없이 파일 하나(`eagle_taxi.db`)에 다 들어감. 학교 프로젝트 MVP라 충분. 명세서·개발 계획안에 "절대 변경 금지"로 명시.

### 5-3. JWT 토큰은 뭔가?
로그인 성공 시 백엔드가 "이 사람 진짜 OO다"라고 서명한 암호화된 출입증. 프론트는 이걸 localStorage에 저장. 다른 API 호출할 때 `Authorization: Bearer <토큰>` 헤더로 보냄. 백엔드는 서명 검증해서 누군지 확인 (DB 매번 조회 안 해도 됨).

### 5-4. 왜 bcrypt 4.0.1로 버전 못 박았나?
bcrypt 5.0이 너무 최신이라 passlib 1.7.4(우리 비밀번호 라이브러리)와 호환 안 됨. 해결: bcrypt를 4.0.1로 다운그레이드. `requirements.txt` 그대로 설치하면 안전, 함부로 업그레이드 X.

### 5-5. 왜 422가 아니라 400을 쓰나?
기능명세서가 예외 응답을 "400 + 한글 메시지"로 정의. FastAPI 기본은 422 + 영문 구조화. 우리는 글로벌 핸들러로 422 → 400 + 한글 변환. 프론트는 `response.data.detail` 그대로 alert 띄우면 됨.

### 5-6. 시간대(KST) 처리
명세서: "기본 시간 기준 KST". 프론트가 `"2026-05-13T18:00:00"`(timezone 없는 ISO)으로 보내면 → KST 18시로 해석. 백엔드 현재 시각도 KST 기준 비교. 캐비엇: NOTES.md 1번 항목에 기록.

## 6. 보안 검토

### 6-1. 잘 막힌 것 (안전)
| 위협 | 우리 방어 |
|---|---|
| SQL 인젝션 | SQLAlchemy ORM 파라미터 바인딩 — 자동 이스케이프 |
| 비밀번호 평문 노출 | bcrypt 해시 + UserOut 스키마에 password_hash 필드 없음 |
| 토큰 위조 | JWT HS256 서명. JWT_SECRET_KEY 모르면 위조 불가 |
| 계정 enumeration | 로그인 실패 시 메시지 동일("이메일 또는 비밀번호가...") |
| XSS | 백엔드는 JSON만 반환, HTML 안 그림. 프론트 React가 자동 이스케이프 |
| CSRF | JWT를 Authorization 헤더로 보냄 (쿠키 X) → 구조적으로 회피 |
| Mass Assignment | Pydantic이 정의된 필드만 받음. role 같은 건 서버에서만 결정 |
| 입력 검증 | Pydantic + DB CHECK 제약 이중 방어 |

### 6-2. 약한 부분 (MVP 인정)
| 위협 | 현재 상태 | 권장 |
|---|---|---|
| Rate limiting | 없음 | MVP라 그대로 OK. 실서비스 시 slowapi 도입 |
| JWT_SECRET_KEY 기본값 | `"dev-secret-change-me"` | 운영 시 `.env`에 랜덤 키 명시 필수 |
| 비밀번호 정책 | 최소 8자만 | MVP OK |
| 토큰 무효화 (로그아웃) | 불가 (JWT 한계) | 실서비스 시 Redis blocklist |
| HTTPS | 없음 (로컬만) | 시연 OK, 실서비스 시 필수 |
| 에러 응답 정보 노출 | `errors` 배열에 Pydantic 디테일 포함 | 운영 시 끄거나 줄이기 |

### 6-3. 종합 의견
명세서 자체에도 "MVP 개발 기준이며 실제 서비스화 단계에서는 보안 검토 필요"로 명시되어 있음 (Section 7). 발표 + 같은 Wi-Fi 시연 기준이면 현재 수준으로 충분.

## 8. 예상 질문 & 답변

### PM(용석희) 측

Q1. "지금 회원가입·로그인 진짜 다 되는 거 맞아?"
- 네, 자동 테스트 27건이 다 통과합니다. Swagger UI(`http://localhost:8000/docs`)에서 즉시 시연 가능합니다.

Q2. "Key 받으면 바로 요금 표시 되는 거지?"
- 네. `.env` 파일에 한 줄만 추가하면 다음 파티 생성부터 실제 요금이 들어옵니다. 코드는 변경 안 합니다.

Q3. "AI 코드 그대로 써도 안전한가?"
- 27건 자동 테스트로 회귀 방지. 모든 검증을 Pydantic + DB CHECK로 이중 깔아둠. 매직 스트링은 `constants.py`에 한 곳으로 모아 변경 시 한 줄 수정으로 끝남. 캐비엇은 NOTES.md에 누적.

Q4. "다음 주 일정 빡빡한데 채팅까지 갈 수 있어?"
- 검색/추천/취소 4개 API는 다음 주 전반부에 마무리 가능. 채팅(WebSocket)은 후반부 진입. 채팅 UI는 심지수님 작업량과 맞물려 일정 협의 필요.

Q5. "API 변경 절차는?"
- 개발 원칙대로 PM·TL·QA&Front 3자 합의 후 명세서 갱신 → 코드 반영. TL이 명세서 변경 없이 임의로 엔드포인트 바꾸지 않음.

Q6. "오늘 새로 갱신된 API 명세서 반영했나?"
- 오타 수정 등 작은 변경은 이미 일치. 신설 섹션(공통 에러, DB 테이블 기준, 상태 코드 세분화)은 회의 결정 후 반영 예정. 일부는 우리 구현이 이미 명세대로 (cancel_reason, party_gender, fare_source 등 컬럼).

### QA&Front(심지수) 측

Q7. "회원가입 응답 어떻게 와?"
- `{ "message": "회원가입이 완료되었습니다.", "user": { id, email, name, gender, role, created_at } }`. message는 alert 띄우고, user는 로컬 상태 저장.

Q8. "로그인 토큰은 어떻게 받고 어디에 저장해?"
- `POST /api/auth/login` 응답에 `access_token` 있음. localStorage에 `access_token` 키로 저장. 이후 모든 API 호출 시 헤더에 `Authorization: Bearer <token>` 붙이면 됨.

Q9. "에러 메시지는 어디서 꺼내?"
- 모든 에러 응답이 `{ "detail": "한글 메시지" }` 형식. 그대로 alert 띄우면 됨. 영문 파싱 코드 안 짜도 됨.

Q10. "출발 시간은 어떤 형식으로 보내?"
- ISO 8601 문자열. `"2026-05-13T18:00:00"` 같은 형식. timezone 안 적으면 KST로 해석. JavaScript의 `date.toISOString()` 결과(UTC)로 보내도 백엔드가 알아서 KST로 변환.

Q11. "지도에 마커 찍을 좌표는?"
- 파티 상세(`GET /api/parties/{id}`) 응답에 `start_lat`, `start_lng`, `end_lat`, `end_lng` 4개 포함.

Q12. "토큰 만료되면 어떻게 알아?"
- API가 `401 + { "detail": "로그인 정보가 만료되었습니다." }` 응답. 받으면 로그인 화면으로 라우팅.

Q13. "파티 정원 다 차면 어떻게 표시?"
- 응답의 `status` 필드가 `"matched"`로 자동 변경됨. 추가 참여 시도하면 400 에러로 "모집 중인 파티가 아닙니다."

### OM(최우진) 측

Q14. "알고리즘 설계서의 매칭 점수 계산 들어가 있어?"
- 아직요. 검색(F-PARTY-007)·유사 추천(F-PARTY-008) 다음 주 작업 때 들어갑니다. 점수표(출발지+30 / 도착지+30 / 시간±10분+25 등) 그대로 구현 예정.

Q15. "테스트 데이터 어떻게 확인할 수 있어?"
- Swagger UI에서 직접 만들 수 있음. 보고서용 데이터 필요하면 별도 시드 스크립트 만들 수 있음.

### 공통

Q16. "발표 시연은 어떻게 해?"
- 발표자 PC에서 `uvicorn app.main:app --reload` 한 줄로 백엔드 켜고, 프론트 dev 서버 켜고, 같은 Wi-Fi 다른 노트북에서 접속. CORS 설정 그날 조정.

Q17. "혹시 갑자기 백엔드가 안 켜지면?"
- 의존성 설치(`pip install -r requirements.txt`)가 최신 기준이고 Python 3.11/3.12로 동작 확인 완료. 안 되면 백업으로 TL 노트북에서 실행 가능.

Q18. "코드 한 번 보고 싶은데?"
- GitHub push는 회의 결과 반영 후 진행 예정. push 권한·브랜치 전략 확정되면 바로 올림.

Q19. "보안은 괜찮아?"
- SQL 인젝션·비밀번호·토큰 위조 등 주요 위협은 구조적으로 방어. Rate limiting 같은 실서비스용 보강은 안 했지만 MVP 범위에서는 충분. 명세서도 "실서비스 시 보안 검토 필요"로 명시.

Q20. "용어가 어려운데?"
- 자료 섹션 1에 용어 사전 정리. 회의 중 모르는 용어 나오면 그쪽 참고.
